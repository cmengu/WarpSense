# create-issue-autonomous

You are a senior engineer capturing a development issue. Your job is to produce a complete, structured issue specification that gives the exploration phase everything it needs — with zero follow-up questions required.

---

## Your Output Contract

Produce exactly these sections. No extras, no omissions. Be dense and specific — no filler.

---

### 1. Title
`[Type] Specific description under 80 chars`

Types: `Bug` `Feature` `Improvement` `Refactor` `Performance` `Security` `A11y`

---

### 2. TL;DR
3–5 sentences. Cover: what, why it matters, root cause, desired outcome, effort size. No jargon. Quantify impact if possible (% users affected, time saved, revenue impact).

---

### 3. Root Cause Analysis
Answer "why" 5 levels deep. State the surface problem, then each underlying cause. End with the true root cause in one sentence.

---

### 4. Current State

**What exists today:**
List every relevant file, component, API endpoint, and data model with one-line descriptions. Include file paths. Be exhaustive — if it touches this issue, list it.

**What's broken or missing:**
For each gap: what the user wants to do, what currently happens, and why it fails. Include workarounds users are using now.

---

### 5. Desired Outcome

**User flow after fix:**
Step-by-step: what the user does → what the system does → what the user sees. Cover the primary flow and at least 2 edge flows (error state, empty state).

**Acceptance criteria:**
List 8–12 specific, testable criteria. Format: `User can [action]` or `System does [behaviour]`. Each must be verifiable — no vague criteria like "works correctly."

**Out of scope:**
List at least 3 things explicitly excluded, with a one-line reason each.

---

### 6. Constraints

Cover all that apply:
- Tech stack requirements (must use / must not use)
- Performance targets (response time, bundle size, memory)
- Browser / device / environment support
- Accessibility standard (WCAG level)
- Timeline or deadline
- Blocked by / blocks (other issues)
- Approval required from

---

### 7. Risks
List 5–8 risks. For each: description, probability (Low/Med/High), impact (Low/Med/High), mitigation. Focus on risks that could block execution or cause rework.

---

### 8. Open Questions
List every unresolved decision. For each: the question, your current assumption, confidence in that assumption (Low/Med/High), and who can resolve it. Mark blockers with 🔴.

---

### 9. Classification
- **Type:** bug / feature / improvement / refactor / performance / security / a11y
- **Priority:** P0 (production broken) / P1 (high impact) / P2 (standard) / P3 (nice-to-have)
- **Effort:** XS (<4h) / S (4–8h) / M (8–16h) / L (16–40h) / XL (40–80h)
- **Effort breakdown:** Frontend _h + Backend _h + Testing _h + Review _h = Total _h

---

## Output Rules

- Write every section. If a section has nothing, write "N/A — [brief reason]" rather than omitting it.
- File paths must be real (explore the codebase if needed to confirm them).
- Acceptance criteria must be testable — someone should be able to check each one without asking you.
- Risks must be specific to this issue, not generic boilerplate.
- Do not include planning or implementation details — that is the exploration phase's job.
- Target length: 600–900 words of substantive content. No padding.

# Task Description:

Threshold Configuration Admin UI
The Demo Line
"We calibrate to your spec sheet" is worth more in a sales conversation than any feature you can build. This is the plan that turns a prototype into a product a procurement team can evaluate.
Step 1: Data Model
typescript// types/thresholds.ts
interface WeldTypeThresholds {
  weld_type: string;        // "mig" | "tig" | "stick" | "flux_core"
  angle_target_degrees: number;
  angle_warning_margin: number;   // ±n degrees = yellow
  angle_critical_margin: number;  // ±n degrees = red
  thermal_symmetry_warning_celsius: number;  // max |N-S| or |E-W|
  thermal_symmetry_critical_celsius: number;
  amps_stability_warning: number;  // max std deviation
  volts_stability_warning: number;
}
```

### Step 2: Backend — thresholds table + routes

New table `weld_thresholds` with `weld_type` as PK. Seed with sensible defaults:

| Weld Type | Angle Target | Warning | Critical |
|-----------|-------------|---------|----------|
| MIG | 45° | ±5° | ±15° |
| TIG | 75°10° |
| Stick | 20° | ±8° | ±20° |
| Flux Core | 45° | ±7° | ±18° |

Two endpoints: `GET /api/thresholds` (all) and `PUT /api/thresholds/:weld_type` (update one). That's ~25 lines of Python.
```

Step 4: Wire thresholds into scoring
In backend/services/thermal_service.py and scoring logic, replace hardcoded constants (45, 5, 15, 20) with a get_thresholds(weld_type) call. Cache it — thresholds change rarely. This is the connection that makes configuration real, not cosmetic.
Step 5: Show active threshold on WelderReport
Small callout below the score: "Evaluated against MIG spec — Target 45° ±5°". One line, fetched with the session. This tells the supervisor "this score means something specific to your operation."
