# WarpSense — One-Click Docker Deploy

Deploy the full stack (PostgreSQL, Backend, Frontend) in ~5 minutes.

## Prerequisites

- **Docker** (20.0+ recommended)
- **Docker Compose** (V2 plugin or standalone `docker-compose`)
- **Minimum 2GB RAM** recommended (containers may fail on lower memory)

```bash
# Install Docker (Linux example)
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
newgrp docker
```

## Quick Start

```bash
# 1. Clone and enter repo
git clone https://github.com/yourname/shipyard-welding
cd shipyard-welding

# 2. Deploy (generates DB password, builds, starts, seeds demo data)
chmod +x deploy.sh
./deploy.sh

# 3. Access
# Frontend:  http://localhost:3000
# Demo:      http://localhost:3000/demo
# Backend:   http://localhost:8000/docs
```

## Environment Variables

Copy `.env.example` to `.env` and customize before running `./deploy.sh`:

| Variable | Description | Default |
|----------|-------------|---------|
| `DB_PASSWORD` | PostgreSQL password | Auto-generated by deploy.sh |
| `NEXT_PUBLIC_API_URL` | API URL for browser (frontend build) | `http://localhost:8000` |

For remote deployment, set `NEXT_PUBLIC_API_URL=http://your-server-ip:8000` so the frontend can reach the backend from the user's browser.

## Verify Deployment

```bash
docker compose ps
# All services should show "Up (healthy)"

# View logs
docker compose logs -f

# Test API
curl http://localhost:8000/health
curl http://localhost:8000/api/sessions/sess_expert_001 | head -c 200
```

## Stop / Remove

```bash
# Stop services
docker compose down

# Stop and remove all data (CAUTION)
docker compose down -v
```

## Architecture

```
Host Ports          Containers
3000  →  shipyard_frontend (Next.js)
8000  →  shipyard_backend (FastAPI + Alembic)
5432  →  shipyard_postgres (not exposed by default; add to docker-compose for debugging)
```

PostgreSQL data persists in the `postgres_data` volume across restarts. To expose Postgres for debugging (psql, DBeaver), add under the `postgres` service in `docker-compose.yml`:

```yaml
ports:
  - "5432:5432"
```

## Risks & Mitigation

| Risk | Mitigation |
|------|-------------|
| Ports 3000/8000 in use | Deploy script exits with error; free ports or stop conflicting processes |
| Old Docker version | Require Docker 20.0+ (see Prerequisites) |
| Migrations fail on existing DB | `alembic upgrade head` is idempotent (safe to re-run) |
| Network firewall | Open ports 3000, 8000 in firewall for remote access |

## Troubleshooting

| Symptom | Cause | Resolution |
|---------|-------|------------|
| **Port already in use** | Another process uses 3000 or 8000 | Script exits with error. Run `lsof -i :3000` / `lsof -i :8000` to find the process, stop it, or change ports in `docker-compose.yml`. |
| **CORS errors** | Frontend in browser can't reach backend | Set `CORS_ORIGINS` in `.env` to include your origin (e.g. `http://your-server-ip:3000`). For remote deploy, also set `NEXT_PUBLIC_API_URL=http://your-server-ip:8000` before running `./deploy.sh`. |
| **Seeding failed** | DB not ready, schema mismatch, or import error | Check logs: `docker compose logs backend`. Error output is printed; fix the root cause. Deployment continues without demo data. |
| **Services fail to start** | DB init slow, insufficient RAM, or image pull failure | Wait longer or check `docker compose logs`. Ensure 2GB+ RAM. Run `docker compose pull` then retry. |
| **Health check timeout** | Backend/Frontend slow to start | Increase `wait_for_health` timeout in `deploy.sh` or check backend logs for migration/startup errors. |
